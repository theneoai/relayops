# 自动驾驶功能安全审查清单 (ISO 26262 ASIL-D)

## 1. 安全文化

- [ ] 安全责任人已明确指定
- [ ] 安全相关变更必须经过安全评审
- [ ] 安全事件必须记录和分析
- [ ] 安全培训每年至少一次

## 2. 危害分析与风险评估 (HARA)

### 2.1 场景识别
- [ ] 所有运行设计域(ODD)已识别
- [ ] 所有危害场景已识别（HAZOP方法）
- [ ] 危害事件的严重度(S)、暴露率(E)、可控性(C)已评估
- [ ] ASIL等级已确定

### 2.2 典型危害场景
| 危害 | S | E | C | ASIL |
|-----|---|---|---|------|
| 非预期加速 | S3 | E4 | C3 | D |
| 制动失效 | S3 | E4 | C3 | D |
| 转向失控 | S3 | E4 | C3 | D |
| 目标漏检（碰撞）| S3 | E4 | C2 | D |
| 非预期车道偏离 | S2 | E4 | C2 | C |
| 误触发AEB | S1 | E4 | C3 | A |

## 3. 软件安全要求

### 3.1 编码规范
- [ ] 使用MISRA C++:2008或AUTOSAR C++14
- [ ] 禁止使用动态内存分配（new/delete/malloc/free）
- [ ] 禁止使用异常处理（throw/try/catch）
- [ ] 禁止使用递归调用
- [ ] 禁止使用goto语句
- [ ] 所有变量必须初始化
- [ ] 数组访问必须边界检查

### 3.2 实时性
- [ ] 所有任务有明确的截止时间(WCET)
- [ ] 关键任务（控制循环）周期 ≤ 20ms，抖动 < 1ms
- [ ] 无阻塞调用
- [ ] 无动态内存分配导致的非确定性延迟
- [ ] 调度策略经过分析（Rate Monotonic或EDF）

### 3.3 故障检测
- [ ] 范围检查：所有传感器输入有有效范围验证
- [ ] 合理性检查：交叉验证多个传感器
- [ ] 时序监控：检查数据时效性
- [ ] 看门狗机制：检测任务挂起
- [ ] 心跳机制：检测节点失联
- [ ] 内存保护：MPU/MMU启用

### 3.4 故障处理
- [ ] 单点故障不会导致危害事件
- [ ] 故障安全状态(fail-safe)已定义
- [ ] 故障运行状态(fail-operational)已定义（如适用）
- [ ] 降级策略已定义
- [ ] 故障恢复策略已定义

## 4. 架构安全

### 4.1 冗余设计
- [ ] 安全关键功能有冗余实现
- [ ] 冗余通道相互独立（多样性设计）
- [ ] 比较监控器(Comparator Monitor)已部署
- [ ] 故障注入测试已执行

### 4.2 独立性
- [ ] 不同ASIL等级的软件组件已分离
- [ ]  Freedom From Interference (FFI) 已分析
- [ ] 内存分区已实施
- [ ] 时间分区已实施

## 5. 测试要求

### 5.1 单元测试
- [ ] 覆盖率：语句覆盖 ≥ 100%
- [ ] 覆盖率：分支覆盖 ≥ 100%
- [ ] MC/DC覆盖 ≥ 100%（ASIL-D）
- [ ] 边界值分析已完成
- [ ] 等价类划分已完成

### 5.2 故障注入测试
- [ ] 传感器数据注入异常值
- [ ] 传感器数据丢失模拟
- [ ] 通信延迟/丢包模拟
- [ ] CPU负载过高模拟
- [ ] 内存 corruption 模拟

### 5.3 SIL/HIL测试
- [ ] 所有危害场景有对应测试用例
- [ ] 场景覆盖率 ≥ 95%
- [ ] 边界场景已覆盖
- [ ] 回归测试已执行

## 6. 工具链与流程

- [ ] 编译器已认证（符合ISO 26262工具要求）
- [ ] 静态分析工具已配置（Polyspace/Klocwork）
- [ ] 需求追溯矩阵已建立
- [ ] 配置管理已实施
- [ ] 变更影响分析已执行

- [ ] 所有blocker级别问题已修复
- [ ] 安全测试已全部通过
- [ ] 安全文档已更新
